\section{Backend unit and integration tests}
\label{sec:backend-tests}

The backend test project mirrors the production code's feature folders and is composed of two complementary layers. The \emph{unit-test layer} exercises pure business logic in isolation, driving components such as the deterministic image compositor directly through inputs and outputs without bringing up the rest of the application. Unit tests of this shape are reserved for code whose behaviour can be specified entirely by data.

The \emph{integration-test layer} brings up the ASP.NET Core application in-process and drives it through HTTP, with one xUnit test class per endpoint area covering authentication, refresh-token rotation, hybrid catalogue generation, print export and the credit ledger. The harness applies the schema migrations, runs the real routing and authorisation pipeline, and is deliberate about what it does and does not mock: external AI providers are stubbed at the \texttt{IImageProvider} seam and the recommendation engine's language-model calls are routed through a stub kernel that returns canned outputs for canned inputs, while the database, EF Core and pgvector layers are exercised end to end. The substitution makes the suite reproducible and free to run, while leaving prompt construction, embedding storage, ledger debits and job scheduling fully exercised.

\section{Frontend unit tests}
\label{sec:frontend-tests}

The React Native client is tested with Jest under the Expo preset, so that the same Babel pipeline used at runtime is reused in tests. The suite covers the client's pure utility modules (string formatting, currency display, validation helpers and the small adapters that sit above the API client) through input-output assertions, and grows on every pull request that introduces a new helper. Because the API client and the React contexts that wrap it expose a single side-effect seam, the components that consume them remain trivially mockable in Jest when component-level tests are added.

\section{End-to-end tests with Playwright}
\label{sec:client-tests}

The web build of the React Native client is exercised by Playwright against the Expo web bundle, with the entire backend network surface mocked in-process so that no backend, database, blob account or AI provider is ever contacted and no real generation cost is incurred during a test run. The suite walks the user-visible journeys end to end: account creation through the three-stage onboarding flow, product creation with background removal and search-by-photo retrieval, generation of one announcement, one catalogue and one wallpaper under both fully generative and hybrid modes, A4 print export, and the daily-recommendation handoff into the asset-generation pipeline. The credit ledger is observed throughout, so that the wallet balance after any sequence of operations always matches the sum of the corresponding ledger entries.

Because the React Native web export is a real DOM tree and the iOS/Android builds reuse the same components, a passing Playwright run is strong evidence that the route tree, the state-context layer and the API client are correct on all three surfaces. Native-only behaviour (haptics, OS deep links) is verified manually before each release.

\section{LLM output evaluation with DeepEval}
\label{sec:llm-eval}

The recommendation engine's writer stage produces stochastic text and cannot be verified by equality assertions. It is instead evaluated offline using \emph{DeepEval}~\cite{deepeval2024}, an open-source evaluation framework for language-model applications. The framework was chosen over hand-rolled scripts because its metric library directly covers the axes the platform cares about (answer relevancy, faithfulness, hallucination, and rubric-style judgements over tone and brand voice), and because it reports bootstrap confidence intervals and pass-fail thresholds without requiring bespoke statistics code. The fact that it integrates with the same test runner the rest of the suite already uses was a further point in its favour.

The reference set against which DeepEval is run is a frozen collection of $120$ $(\text{shop}, \text{day}, \text{signal})$ tuples that span urban and rural shop profiles, weak and strong context signals, and both supported languages. Three complementary checks are run on every model or prompt change:

\begin{itemize}
    \item Schema and format gate: every output must parse as the declared JSON schema, populate every required field and respect the fixed enumerations for \texttt{type} and \texttt{tone}. Schema failures fail the suite.
    \item Forbidden-phrase regression: a curated denylist of the boilerplate marketing tropes the writer prompt is built to discourage (``discover our premium selection'', ``don't miss out'', and their Romanian equivalents) is matched against the \texttt{title}, \texttt{body} and \texttt{suggestedPost} fields. The suite tracks the per-100-ideas rate of denylist hits and treats a regression in that rate as a blocker for the writer model or prompt change under evaluation.
    \item DeepEval metric battery: each output is scored on answer relevancy, faithfulness and hallucination using DeepEval's built-in metrics, and on two custom rubric metrics, Tone Match and Schema Compliance, whose criteria are written against the platform's brand-voice specification. The judge model is rotated between Claude and Gemini across runs to control for judge bias, following the LLM-as-judge protocol of Zheng et al.~\cite{zheng2023llmjudge}. DeepEval reports mean scores per axis with bootstrap confidence intervals, and the suite flags regressions of more than $0.2$ rubric points.
\end{itemize}

\subsection*{Cross-model evaluation}

The same DeepEval harness is used to compare candidate writer models head-to-head on the reference set, so that decisions about which backbone to host, fine-tune or call as a managed service can be made on evidence rather than intuition. Five models are evaluated: Gemma 3 4B as the open-weights base, Gemma 3 4B (fine-tuned) after supervised fine-tuning on the platform's brand-voice corpus, DeepSeek as an open-weights frontier comparison, and the closed Claude and ChatGPT families as managed-service baselines. The reading of Figure~\ref{fig:deepeval-models} is the story the chapter argues. The off-the-shelf Gemma 3 4B model is the weakest on every axis, with the largest gap on Tone Match, the metric most sensitive to brand voice and to Romanian-language register. Supervised fine-tuning on the platform's brand-voice corpus closes most of that gap: the fine-tuned variant is within two points of the strongest model on every axis and leads on Tone Match ($93.8$), overtaking both Claude ($90.6$) and ChatGPT ($88.4$). The closed-frontier models retain a small lead on raw Faithfulness and Hallucination-Free Rate, where their scale advantage shows. On the operational dimensions that matter for a small-retailer product, namely tone match and schema compliance, a fine-tuned $4$-billion-parameter open model is at parity with managed services that cost two orders of magnitude more per call, and is therefore the platform's default writer.

A complementary human-rated pass is run before each release: two independent reviewers perform pairwise comparisons of the platform's output against the un-adapted seven-billion-parameter writer on a $30$-tuple stratified sub-sample, and the win-rate is recorded. Image generation is evaluated only manually, by running a small canonical prompt set against each live provider and inspecting the assets for product fidelity and tonal fit.

\section{Continuous integration and the merge gate}
\label{sec:ci-merge-gate}

The build pipeline is hosted on GitHub Actions. On every pull request it runs three automated gates: the backend integration suite against an ephemeral PostgreSQL container with pgvector enabled, the Jest frontend suite, and the Playwright end-to-end suite against the in-process-mocked web build. All three must pass before merge into \texttt{master}, and the gate has no manual override. The DeepEval evaluation of Section~\ref{sec:llm-eval} is deliberately left out of this per-pull-request pipeline, because it drives live writer models and an LLM judge and therefore needs provider credentials and a non-trivial call budget; it is instead run offline on recommendation model or prompt changes rather than on every commit.
